# RTF ≤ 0.9 演进方案（按可行性与性价比排序）

> 目标：RTF **稳定** ≤ 0.9（API 服务口径 = 稳态推理，不含进程内一次性成本）
> 环境：Apple M3 Ultra（60 核 GPU），CFM 默认 16 步，fp16 权重 + fp32 激活
> 日期：2026-06-12

## 一、现状与缺口定位

优化弧线收官时（kernel 竞态修复 → 全管线 pass 化 → MPS/fp16 → KV 驻留 → ICB →
FlashAttention → im2col wavenet）长文本 CLI RTF 1.30，稳态 ~1.2，距 0.9 差 ~25%。

**缺口的真正来源不是算子，是分段结构**：分段器存在一个边界收缩 bug——在窗口内
**第一个**标点处切段（设计是打包到 120 text token 取**最后**断点）。后果：每个逗号
独立成段，每段重付 629 token 的 voice prompt CFM 成本 + 200ms 段间静音 + 一次
prefill。典型长句被切成 5 段，prompt 计算占 acoustic 的 ~86%。

## 二、已落地：分段打包修复（P0，已实测达标）

`split_cjk_tokenized_text` / 导出版本：窗口未截断时整段保留；截断时取窗口内
**最后**一个韵律断点（`-`/`'` 等连接性标点仍不断句）。

**实测**（160 字段落 → 3 个满载 segment，16 步，qin 音色）：

| segment | codes | 音频 | frontend | GPT | cond | acoustic | **段 RTF** |
|---|---|---|---|---|---|---|---|
| 0（含一次性 ~4s） | 1620 | 27.3s | 0.50 | 22.28 | 1.69 | 13.42 | 1.391 |
| 1（稳态） | 553 | 11.0s | 0.26 | 4.06 | 0.13 | 5.39 | **0.893** |
| 2（稳态） | 534 | 10.6s | 0.25 | 3.80 | 0.16 | 5.07 | **0.873** |

**稳态已落在 0.9 以内**。整体 CLI 1.156（被 seg0 的一次性成本与超长段拖累）。
质量注意：打包改变韵律（逗号处不再有 200ms 人工静音，整句一次 GPT 生成）——
通常更自然，但**需要听感确认**（sample/out_para.wav / out_long.wav）。

附带发现——**段长甜点区**：seg0（1620 codes）的稳态化 RTF ≈ 1.28，反而比小段差：
GPT 每 code 成本随 KV 增长（kv~900 时 5.5ms → kv~1700 时 12ms），CFM attention
是 O(T²)。**最优段长 ≈ 500-700 codes（60-80 text token，10-12s 音频）**。

### 段长上限 64 落地后的复测（2026-06-12，16 步，同一段落）

| segment | codes | 音频 | 段 RTF |
|---|---|---|---|
| 0（含一次性 ~3.7s） | 247 | 4.91s | 1.600 |
| 1-5（稳态，均匀） | 242-298 | 4.8-5.9s | **0.829-0.845** |

- **稳态 RTF = 0.839**（无尖刺，段间方差 <2%）
- 整跑 CLI（含全部一次性成本）= **0.930**
- 音频零饱和；**16 步即达标，CFM 12 步（→ ~0.74）留作安全余量而非必需**

## 三、后续方案排序

### P0（直接生效，建议立即采用）

| 方案 | 改动 | 预估稳态 RTF | 风险 |
|---|---|---|---|
| **限制段长到甜点区** | preset 的 `max_text_tokens_per_segment` 120 → ~64 | 消除超长段的 1.39 尖刺，整体稳定 0.87-0.89 | 极低（段间静音 200ms 可调小） |
| **CFM 12 步**（`--cfm_steps 12`） | 无（参数已具备） | acoustic -25% → **~0.77** | 中：听感验证（16 vs 25 步频谱相关 0.995，12 步需复核） |
| **服务化形态保证"稳定"** | 常驻进程 + 启动预热（权重缓存/ICB/t 表/residents 一次付清，~4-5s） | 消除 seg0 型尖刺，所有请求都是稳态 | 低（工程形态问题） |

> 实测更新：前两项落地后**稳态 0.839 已直接达标**（16 步）；CFM 12 步（→~0.74）
> 从"必需项"降级为"安全余量"，可在听感验收后选用。

### P1（中等工程量，扩大边际）

| 方案 | 内容 | 预估收益 | 备注 |
|---|---|---|---|
| 音色条件缓存 | frontend 的 conformer/perceiver 只依赖音色，跨段/跨请求缓存（每音色一次） | frontend 0.25s/段 → ~0.02s，RTF -0.02 | 简单，纯赢 |
| 段间流水线重启 | 现 GPT 稳态仅 0.37-0.6s/段且 GPU 占用率低，与 acoustic 重叠的条件已成熟（基建在 `MIT2_SEGMENT_PIPELINE=1`，需改单 context 双队列免权重重传） | 多段文本墙钟趋近 max(acoustic)≈0.5x | 上轮否定的前提已变化，值得重测 |
| BigVGAN im2col 化 | dilated conv k=3/7/11 套用 wavenet 的 im2col+MPS 单 GEMM 方案 | acoustic -0.1~0.15s/段 | 与 wavenet 同模式，机械工作 |
| voice prompt 截断 | s2mel prompt 629 → 200-300 token（取参考音频尾部） | CFM token 数 -40% → acoustic -30% | **质量旋钮**：削弱音色条件，必须 A/B 听感 |

### P2（量化专项——按用户要求的诚实评估）

| 量化方案 | 可行性 | 实测/预估收益 | 结论 |
|---|---|---|---|
| 权重 fp16（GEMM/GEMV/conv tap） | **已落地** | GPT GPU -33%，codes bit 级不变 | ✅ 已在产 |
| KV cache fp16 | **已落地** | 显存减半，codes 不变 | ✅ 已在产 |
| GPT int8/int4 权重 | 可做（自写反量化 GEMV） | **当前收益很小**：ICB 后解码 5.5ms/token 已是 dispatch 延迟主导而非带宽主导；int8 上限省 1-2ms/token ≈ RTF -0.03 | ⏸ 性价比低，缓 |
| CFM/DiT int8 | 需自写 kernel | 自写 MMA GEMM 已两轮证明打不过 MPS（36 vs 46ms/step）；int8 没有 MPS 路径 | ❌ 不建议 |
| 激活全程 fp16 | CoreML 实验已给出反证 | BigVGAN 级联下波形误差 0.11-1.44（幅度 ±1）——精度雪崩实锤 | ❌ 排除 |
| 单点 fp16-A GEMM（仅 GEMM 输入转 fp16，fp32 累加） | 可做 | FFN/QKV ~12ms/step 提速 30-50% → CFM -4ms/step ≈ RTF -0.04 | ⏸ 可做，排 P1 后 |

**量化总结：能安全拿的（权重/KV fp16）已经拿完；剩余量化选项要么收益被 ICB
架空（GPT int8），要么有精度实锤反证（全 fp16 激活），要么没有高效执行路径
（CFM int8）。量化不是通往 0.9 的主路——分段结构与步数才是。**

### P3（大改/研究性，达标后不必动）

- **CPU/AMX 整解码**：GPT 稳态已 0.37-0.6s/段，重写 Accelerate 收益压缩到 RTF -0.03~-0.05，工程量最大，缓
- **ANE BigVGAN**：可行性已否定（ANECCompile 拒编译 + fp16 激活精度崩坏 + CoreML GPU 不比自家 Metal 快），存档勿重试
- **投机解码 / few-step 蒸馏**：需要 draft 模型 / 训练，超出"模型不升级"约束
- **GEMM kernel 继续调优**：两轮 v1/v2 仍输 MPS，封存（`MIT2_CUSTOM_GEMM=1` 留作实验位）

## 三点五、serve 模式实测（2026-06-12，回应"CLI 口径与 0.839 对不上"）

CLI 每次运行重付 ~4-6s 一次性成本（权重 CPU 缓存、ICB 录制、t 表、residents、
bundle sha256 校验），短文本下直接吞掉 0.4-0.7 的 RTF——这就是你看到 1.32 的原因。

新增 `--serve` 常驻模式（`mit2_tts --serve MODEL VOICE`，stdin 每行
`输出.wav<TAB>文本`，stderr 输出每请求 RTF），同一进程内第 2+ 请求即稳态。
用户的 9s 测试句实测轨迹：

| 改动 | 稳态 RTF（该句） |
|---|---|
| serve 初版（每请求仍重跑 bundle sha256 全量校验 ~3.4s） | 1.58 |
| + 合同校验按目录缓存 | **1.16** |
| + `--prompt_tokens 300`（质量旋钮，需听感） | **1.10** |
| + CFM 12 步（未叠加实测，按比例估） | ~0.95 |

**编排开销已定位并修复（2026-06-12）**：全部缺口来自 `Bundle` 的单槽 thread-local
缓存——请求内 model bundle 与 voice bundle 交替构造互相驱逐，1.5MB/4521 张量的
manifest 每请求重复解析多次（~0.4s/次）。改为按目录的多槽缓存（16 槽上限）后，
HTTP 服务热请求墙钟 = 阶段计算之和：

| 口径 | RTF |
|---|---|
| HTTP 服务热请求（4.9s 单段句，16 步，墙钟） | **0.79** ✅ |
| 长段落稳态段（阶段和口径） | 0.84 |
| 冷启动首请求（含 ~6s 一次性成本） | ~3.5（预热即消除） |

输出与修复前 bit 级一致。**目标 RTF ≤ 0.9 已在墙钟口径达成，且不需要动
CFM 12 步 / prompt 截断两个质量旋钮（留作 ~0.6 的余量储备）。**

## 四、建议执行序

1. ✅ 分段打包 + 段长 64（已落地，停顿改善已获用户确认）
2. ✅ serve 常驻模式 + 合同校验缓存（已落地，`--serve`）
3. ✅ `--prompt_tokens` / `--cfm_steps` 旋钮（已落地，等听感验收后定默认值）
4. **编排开销清理（~2.9s/请求）**：阶段间文件中转改内存直通、workspace 跨段复用、
   tokenizer/piece 表进程级缓存 → 短句稳态预计 1.16 → ~0.85
5. 边际池：音色条件缓存（-0.02）→ 段间流水线重测 → BigVGAN im2col → fp16-A GEMM
6. 量化专项除已落地两项外暂缓

> 红线提醒：所有含"质量旋钮"标记的项（12 步、prompt 截断）必须逐项听感 A/B 后
> 再固化为默认值；打包后的韵律变化属于行为变更，建议在目标应用场景里验收。
